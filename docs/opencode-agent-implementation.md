# OpenCode Agent Implementation

This document explains how Agents Team renders and installs canonical agents for
OpenCode. It is intentionally specific to OpenCode. Codex and Claude Code should
get equivalent implementation notes later, but their behavior should not be
inferred from this file because their native agent models differ.

Official reference: <https://opencode.ai/docs/agents/>

## Goal

Agents Team keeps one canonical set of agents in `agents/` and converts those
files into OpenCode-native Markdown agent files.

The expected OpenCode behavior is:

- `orchestrator` is selectable as a primary agent.
- `builder`, `explorer`, `researcher`, and `reviewer` are sub-agents.
- Sub-agents are not selectable as main agents.
- The primary orchestrator can invoke the allowed sub-agents through OpenCode's
  task permission system.
- Users can still mention visible sub-agents directly with `@agent-name` unless
  a future config marks them as hidden.

## Native OpenCode Contract

OpenCode supports agents as Markdown files with YAML frontmatter.

Global install location:

```text
~/.config/opencode/agents/
```

Project install location:

```text
<project>/.opencode/agents/
```

The filename becomes the OpenCode agent id. For example:

```text
~/.config/opencode/agents/reviewer.md
```

creates an OpenCode agent named `reviewer`.

The frontmatter controls the agent metadata:

```markdown
---
description: Reviews code for correctness, security, regressions, and missing tests.
mode: subagent
model: openai/gpt-5.3-codex
permission:
  edit: deny
  bash: ask
  webfetch: deny
---

You are a reviewer agent.
```

The important OpenCode `mode` values are:

| Mode | Meaning |
| --- | --- |
| `primary` | Selectable as a main OpenCode agent. |
| `subagent` | Available for delegation or mention, but not a main agent. |
| `all` | Both primary and sub-agent behavior. This is also the OpenCode default when no mode is read. |

Because `all` is the default, any rendering bug that prevents OpenCode from
reading `mode: subagent` can make helper agents appear as selectable primary
agents.

## Critical Frontmatter Rule

OpenCode expects the YAML frontmatter to start at the beginning of the Markdown
file.

Generated OpenCode files must start with:

```markdown
---
```

They must not start with an HTML comment, generated marker, blank prose, or any
other content before the frontmatter.

Correct:

```markdown
---
description: ...
mode: subagent
---

<!-- Generated by Agents Team. Do not edit directly. -->

Prompt body...
```

Incorrect:

```markdown
<!-- Generated by Agents Team. Do not edit directly. -->
---
description: ...
mode: subagent
---
```

The incorrect form can cause OpenCode to ignore the YAML metadata. If that
happens, `mode` is not read and OpenCode may treat the agent as `all`, which
makes a sub-agent show up as a primary selectable agent.

This was the root cause of the April 2026 OpenCode issue where generated
sub-agents appeared as main agents.

## Canonical Source

The canonical agent files live in:

```text
agents/
```

Each file is Markdown with YAML frontmatter plus a shared prompt body.

Example source file shape:

```markdown
---
id: reviewer
description: Use proactively for code review focused on correctness, security, regressions, maintainability risks, and missing tests.

targets:
  opencode: true

model:
  opencode: openai/gpt-5.3-codex

permissions:
  read: allow
  search: allow
  edit: deny
  write: deny
  delete: deny
  bash: ask
  network: deny
  webfetch: deny
  mcp: ask

overrides:
  opencode:
    mode: subagent
    permission:
      edit: deny
      bash: ask
      webfetch: deny
---

You are a reviewer agent.
```

OpenCode-specific native settings belong under:

```yaml
overrides:
  opencode:
```

The adapter applies these overrides after deriving the normalized default
fields, so OpenCode overrides win.

## Renderer Implementation

OpenCode rendering is implemented in:

```text
agents_team/installers/opencode.py
```

The adapter class is:

```python
class OpenCodeAdapter:
    tool = "opencode"
```

The filename is generated by `output_name()`:

```python
def output_name(self, agent: Agent) -> str:
    return f"{agent.id}.md"
```

The main rendering flow is `render()`:

1. Apply any OpenCode prompt override to the canonical prompt body.
2. Build required native fields.
3. Default every generated OpenCode agent to `mode: subagent`.
4. Add the OpenCode-specific model if configured.
5. Add mapped permissions from the portable permission model.
6. Merge `overrides.opencode`, allowing native overrides to win.
7. Emit Markdown with YAML frontmatter first.
8. Emit the generated marker after the frontmatter.
9. Emit the prompt body.

The render shape must remain:

```python
return (
    "---\n"
    f"{yaml_frontmatter(data)}\n"
    "---\n\n"
    f"<!-- {GENERATED_MARKER} -->\n\n"
    f"{body.strip()}\n"
)
```

Do not move `GENERATED_MARKER` above the opening `---` for OpenCode.

## Field Mapping

| Canonical Field | OpenCode Output |
| --- | --- |
| `id` | Filename, such as `reviewer.md`. |
| `description` | `description` frontmatter field. |
| Markdown body | Markdown body after frontmatter. |
| `model.opencode` | `model` frontmatter field. |
| `permissions.edit` | `permission.edit`. |
| `permissions.bash` | `permission.bash`. |
| `permissions.webfetch` | `permission.webfetch`. |
| `overrides.opencode` | Merged into frontmatter after normalized fields. |

`reasoning.opencode` is currently documented in the canonical format but is not
emitted by the OpenCode adapter. OpenCode supports provider-specific additional
options, so a future change may map this to a native field such as
`reasoningEffort` if that becomes part of the desired OpenCode contract.

## Mode Policy

The adapter defaults all OpenCode agents to:

```yaml
mode: subagent
```

That default is intentional. Most canonical agents are helpers and should not be
main selectable agents.

The only current canonical primary OpenCode agent is `orchestrator`, configured
in:

```text
agents/orchestrator.md
```

with:

```yaml
overrides:
  opencode:
    mode: primary
```

Any future agent that sets `overrides.opencode.mode: primary` should be treated
as a product decision, not a casual implementation detail. Primary agents change
what users can select as the active OpenCode agent.

If a helper agent appears as a primary agent in OpenCode, check these first:

1. The installed file starts with `---` at line 1.
2. The installed frontmatter contains `mode: subagent`.
3. The agent source did not override `overrides.opencode.mode` to `primary` or
   `all`.
4. OpenCode is reading the expected install directory.

## Task Delegation Policy

OpenCode can control which sub-agents a primary agent may invoke through
`permission.task`.

The orchestrator currently uses a deny-by-default task policy:

```yaml
permission:
  task:
    "*": deny
    builder: allow
    explorer: allow
    researcher: allow
    reviewer: allow
```

The order matters because OpenCode evaluates matching permission rules in order
and the last matching rule wins. The broad deny rule must come before the
specific allow rules.

This policy means:

- The orchestrator should not see arbitrary sub-agents as available task
  targets.
- The orchestrator can invoke the known specialist team.
- Adding a new sub-agent requires adding a corresponding allow rule if the
  orchestrator should be able to invoke it.

Task permissions do not necessarily hide sub-agents from direct user mention.
They control what the model can invoke through the Task tool. If a future
internal-only sub-agent should not appear in user autocomplete, use OpenCode's
`hidden: true` setting with `mode: subagent`.

## Permission Mapping

The OpenCode adapter maps only the portable actions that have direct OpenCode
permission fields today:

```python
for source, target in (
    ("edit", "edit"),
    ("bash", "bash"),
    ("webfetch", "webfetch"),
):
```

Each mapped value is one of:

```text
allow
ask
deny
```

Example output:

```yaml
permission:
  edit: deny
  bash: ask
  webfetch: allow
```

Other canonical permissions such as `read`, `search`, `write`, `delete`,
`network`, and `mcp` are not currently emitted directly by the OpenCode adapter.
If OpenCode gains or documents native fields for these, update this document,
the adapter, and tests together.

## Install Flow

Installation is coordinated by:

```text
agents_team/installer.py
```

OpenCode install paths are resolved by:

```text
agents_team/paths.py
```

Global OpenCode install:

```python
Path.home() / ".config" / "opencode" / "agents"
```

Project OpenCode install:

```python
project / ".opencode" / "agents"
```

The normal command is:

```bash
uv run agents-team install opencode
```

For a project-local install:

```bash
uv run agents-team install opencode --project <project-path>
```

The installer writes one generated Markdown file per OpenCode-enabled canonical
agent. It does not currently render a separate OpenCode root-instructions file.
OpenCode primary-agent behavior is represented by `mode: primary` in the
orchestrator's own agent file.

## Generated File Safety

Agents Team avoids overwriting manual files unless forced.

The installer checks for:

```text
Generated by Agents Team. Do not edit directly.
```

If an existing file does not contain that marker and is not empty, installation
skips it unless `--force` is used.

For OpenCode, the marker must remain after the YAML frontmatter. It still works
for overwrite detection there because the installer searches the whole file
content, not only the first line.

## Verification Commands

After changing OpenCode rendering or canonical OpenCode overrides, run:

```bash
uv run agents-team validate
uv run pytest
```

After installing locally, verify OpenCode sees the expected modes:

```bash
opencode agent list
```

Expected relevant entries:

```text
builder (subagent)
explorer (subagent)
orchestrator (primary)
researcher (subagent)
reviewer (subagent)
```

You can also inspect the installed files directly:

```powershell
Get-Content $HOME\.config\opencode\agents\builder.md -TotalCount 12
Get-Content $HOME\.config\opencode\agents\orchestrator.md -TotalCount 18
```

The first line should be `---`. For sub-agents, the frontmatter should include:

```yaml
mode: subagent
```

For the orchestrator, the frontmatter should include:

```yaml
mode: primary
```

## Test Coverage

OpenCode renderer tests live in:

```text
tests/test_renderers.py
```

Important tests:

- `test_renders_opencode_markdown`
- `test_renders_opencode_coding_agent_model`
- `test_renders_opencode_orchestrator_as_primary_agent`
- `test_renders_opencode_agents_as_subagents_by_default`

The key regression assertion is:

```python
assert rendered.content.startswith("---\n")
```

That assertion protects the frontmatter parsing contract. Keep it whenever the
OpenCode renderer is changed.

The orchestrator task policy is also covered by asserting the rendered
`permission.task` map includes:

```python
{
    "*": "deny",
    "builder": "allow",
    "explorer": "allow",
    "researcher": "allow",
    "reviewer": "allow",
}
```

If a new sub-agent is added and should be invokable by the orchestrator, update
both `agents/orchestrator.md` and this test.

## Common Failure Modes

### Sub-agents appear as primary agents

Most likely causes:

- The installed Markdown file does not start with YAML frontmatter.
- `mode: subagent` is missing or not inside the frontmatter.
- The canonical agent has an OpenCode override that sets `mode: primary` or
  `mode: all`.
- OpenCode is loading a different global or project-local agent directory.

Recommended checks:

```powershell
Get-Content $HOME\.config\opencode\agents\builder.md -TotalCount 12
opencode agent list
```

### Orchestrator cannot invoke a sub-agent

Most likely causes:

- The sub-agent is missing from `permission.task`.
- The broad `"*": deny` rule appears after the specific allow rule.
- The target sub-agent filename does not match the task permission pattern.
- The sub-agent file is not installed in the OpenCode agent directory being
  loaded.

### Manual file is not updated

Most likely causes:

- The existing file lacks the generated marker.
- The file was manually edited and the installer skipped it.

Recommended options:

```bash
uv run agents-team install opencode --dry-run
uv run agents-team install opencode --force
```

Use `--force` only when overwriting the existing file is intentional.

## Maintenance Checklist

When changing OpenCode support:

1. Update canonical source under `agents/` if the behavior is an agent policy
   decision.
2. Update `agents_team/installers/opencode.py` if the native render contract
   changes.
3. Keep YAML frontmatter as the first bytes of generated Markdown files.
4. Keep the generated marker after the frontmatter.
5. Update `tests/test_renderers.py` for mode, permission, or shape changes.
6. Run `uv run agents-team validate`.
7. Run `uv run pytest`.
8. Reinstall with `uv run agents-team install opencode`.
9. Confirm with `opencode agent list`.
10. Update this document if any native OpenCode behavior or project policy
    changes.

## Future Work

Potential improvements:

- Add validation that only explicitly approved agent ids may use
  `overrides.opencode.mode: primary`.
- Add a canonical role field, such as `role: primary` or `role: subagent`, so
  mode policy is no longer hidden inside `overrides.opencode`.
- Emit OpenCode provider-specific reasoning options if the project standardizes
  on a native field such as `reasoningEffort`.
- Support `hidden: true` for internal sub-agents that should not appear in user
  autocomplete.
- Add a `doctor opencode` check that verifies installed files start with
  frontmatter and that observed `opencode agent list` modes match expectations.
