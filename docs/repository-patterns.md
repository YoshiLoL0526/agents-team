# Repository Patterns

This document describes the proposed project structure and implementation
patterns.

## Proposed Structure

```text
agents/
  orchestrator.md
  explorer.md
  builder.md
  reviewer.md
  researcher.md

agents_team/
  __init__.py
  cli.py
  schema.py
  parser.py
  permissions.py
  renderer.py
  paths.py
  installers/
    __init__.py
    codex.py
    claude.py
    opencode.py

tests/
  test_parser.py
  test_permissions.py
  test_render_codex.py
  test_render_claude.py
  test_render_opencode.py

docs/
```

## Module Responsibilities

### `cli.py`

Owns command parsing, terminal output, and exit codes.

The CLI should remain thin. It should call parser, validator, renderer, and
installer services rather than implementing business logic inline.

### `schema.py`

Defines canonical data structures.

Likely models:

- `Agent`
- `AgentTargets`
- `AgentPermissions`
- `RenderResult`
- `ValidationIssue`

### `parser.py`

Reads Markdown files with YAML frontmatter and returns canonical `Agent` models.

Responsibilities:

- Discover files under `agents/`.
- Parse frontmatter.
- Preserve Markdown body.
- Report syntax errors with file paths.

### `permissions.py`

Owns permission validation and common permission utilities.

Responsibilities:

- Validate action names.
- Validate permission values.
- Detect conflicting permissions.
- Provide adapter helper functions.

### `renderer.py`

Coordinates tool-specific rendering.

Responsibilities:

- Select target adapter.
- Render one or more agents.
- Return generated file names and contents.

### `paths.py`

Resolves global and project-specific installation paths.

Responsibilities:

- Expand `~`.
- Normalize project paths.
- Resolve target directory by tool and scope.

### `installers/`

Adapters for native target formats.

Each adapter should expose a small common interface:

```python
class Adapter:
    tool: str

    def enabled(self, agent: Agent) -> bool: ...
    def output_name(self, agent: Agent) -> str: ...
    def render(self, agent: Agent) -> str: ...
    def validate(self, agent: Agent) -> list[ValidationIssue]: ...
```

## Dependencies

Recommended initial dependencies:

- YAML parser: `PyYAML` or `ruamel.yaml`.
- CLI framework: `typer` or standard library `argparse`.
- TOML output: Python 3.13 has `tomllib` for reading but not writing, so use a
  small TOML writer dependency or implement a constrained writer for Codex files.

Recommended pragmatic choice:

- `typer` for CLI ergonomics.
- `PyYAML` for frontmatter.
- A small internal TOML writer for the limited Codex schema, unless nested Codex
  overrides require a dependency.

## Generated File Marker

Generated files should include a comment marker where the target format allows
comments.

Examples:

```text
# Generated by Agents Team. Do not edit directly.
```

For TOML:

```toml
# Generated by Agents Team. Do not edit directly.
```

For Markdown:

```markdown
<!-- Generated by Agents Team. Do not edit directly. -->
```

This allows future installers to distinguish generated files from hand-written
files.

## Manifests

A future manifest should track installed files.

Suggested location:

```text
~/.agents-team/manifests/
```

Suggested data:

```json
{
  "tool": "codex",
  "scope": "global",
  "installed_at": "2026-04-20T00:00:00Z",
  "files": [
    {
      "agent_id": "reviewer",
      "source": "agents/reviewer.md",
      "target": "~/.codex/agents/reviewer.toml",
      "source_sha256": "..."
    }
  ]
}
```

Manifests are not required for the first renderer implementation but are useful
for reliable update and uninstall commands.

## Testing Strategy

Tests should focus on behavior rather than implementation details.

Initial coverage:

- Frontmatter parsing.
- Required field validation.
- Permission validation.
- Codex TOML rendering.
- Claude Markdown rendering.
- OpenCode Markdown rendering.
- Global and project path resolution.

Installer tests should use temporary directories rather than real home
directories.
